# Принцип определения бот-кандидатов и ограничения сбора данных

Этот документ объясняет, как программа определяет подозрительных пользователей, какие условия используются при построении графа, что происходит при повторном запуске и какие есть ограничения при сборе комментариев из Telegram.

## Краткая логика работы

Проект не использует готовые метки "бот / не бот". Это важно: модель работает как система неконтролируемого обучения. Она не знает заранее, кто бот, а ищет пользователей, поведение которых статистически отличается от большинства.

Пайплайн состоит из четырех этапов:

1. `collector.py` собирает комментарии из Telegram-каналов.
2. `features.py` строит граф совместной активности и рассчитывает признаки по каждому пользователю.
3. `ml_engine.py` ищет аномальных пользователей и группирует их в кластеры.
4. `visualizer.py` строит HTML-графы и аналитический отчет.

Результат нужно трактовать как список подозрительных аккаунтов для дальнейшей проверки, а не как юридическое доказательство автоматизации.

## Какие данные собираются

Для каждого комментария сохраняются:

- `user_id` — ID автора комментария;
- `message_text` — текст комментария;
- `timestamp` — время комментария;
- `post_id` — ID поста канала;
- `reply_to_msg_id` — ID сообщения, к которому относится комментарий;
- `channel` — канал, из которого был собран комментарий;
- `post_timestamp` — время публикации поста.

Сырые данные сохраняются в:

```text
data/raw_comments.csv
```

## Как строится граф совместной активности

В графе:

- узел — это пользователь Telegram;
- ребро между двумя пользователями появляется, если они комментировали один и тот же пост в близкий промежуток времени;
- вес ребра увеличивается, если такая совместная активность повторяется.

Главное условие связи:

```text
два пользователя комментируют один и тот же пост в одном канале с разницей не более 120 секунд
```

Это значение задается в `config.py`:

```python
CO_ACTIVITY_WINDOW_SECONDS = 120
```

Если два аккаунта часто появляются рядом во времени под одними и теми же постами, между ними будет более сильная связь. Такая синхронность является одним из признаков координации.

Граф сохраняется в:

```text
data/graph.graphml
```

Этот файл не предназначен для открытия в браузере как HTML. Его можно анализировать через Python, Gephi или Cytoscape.

## Какие признаки используются для определения подозрительности

Для каждого пользователя рассчитываются базовые поведенческие/графовые признаки и дополнительные категорийные признаки.

### 1. `reaction_speed_mean`

Среднее время реакции пользователя на публикацию поста.

Если аккаунт часто комментирует очень быстро после публикации, это может быть признаком автоматизированного мониторинга или координированной активности.

### 2. `reaction_speed_std`

Стандартное отклонение времени реакции.

Низкое значение может означать, что аккаунт реагирует стабильно и механически. Высокое значение означает более нерегулярное поведение.

### 3. `comment_count_per_day`

Среднее количество комментариев пользователя в день за период наблюдения.

Слишком высокая активность может быть подозрительной, особенно если она сочетается с другими признаками.

### 4. `duplicate_ratio`

Доля повторяющихся или почти повторяющихся сообщений пользователя.

Программа сравнивает тексты через расстояние Левенштейна. Если два сообщения очень похожи, они считаются near-duplicate.

Условие похожести:

```text
Levenshtein distance < 5
```

Это помогает находить аккаунты, которые часто копируют одинаковые или почти одинаковые фразы.

### 5. `graph_degree`

Количество связей пользователя в графе совместной активности.

Высокая степень означает, что пользователь часто появляется рядом с другими пользователями в рамках одного временного окна.

### 6. `clustering_coefficient`

Коэффициент кластеризации в графе.

Он показывает, насколько пользователь находится внутри плотной группы. Для ботоферм важны не только отдельные подозрительные аккаунты, но и группы аккаунтов с плотными взаимными связями.

### 7. `unique_channels`

Количество разных каналов, где был замечен пользователь.

Активность в нескольких каналах может быть нормальной, но в сочетании с высокой скоростью реакции, повторами текста и плотными графовыми связями это становится полезным сигналом.

Также рассчитываются категорийные признаки: разнообразие категорий и доля сообщений пользователя по категориям `18_plus`, `military`, `ipso`, `violence_threats`, `political_agitation`, `spam_scam`.

Матрица признаков сохраняется в:

```text
data/features.csv
```

## Как модель решает, кто бот-кандидат

В `ml_engine.py` используется `IsolationForest`.

Порядок обработки:

1. Берутся числовые признаки пользователя.
2. Пропущенные значения заменяются медианой.
3. Признаки стандартизируются через `StandardScaler`.
4. `IsolationForest` ищет пользователей, которые отличаются от общей массы.

Текущий параметр:

```python
contamination = 0.05
```

Это означает, что модель ожидает примерно 5% аномальных пользователей в датасете.

Пользователь получает:

- `is_bot = 1`, если модель считает его аномальным;
- `is_bot = 0`, если он похож на обычных пользователей.

Важно: `is_bot = 1` в этом проекте означает "бот-кандидат", а не гарантированно доказанный бот.

Результаты сохраняются в:

```text
data/ml_results.csv
```

## Как определяются ботофермы

После поиска аномальных пользователей программа пытается сгруппировать только бот-кандидатов.

Основной метод:

```text
DBSCAN
```

Параметры:

```python
eps = 0.5
min_samples = 3
```

Это значит:

- кластер должен содержать минимум 3 близких бот-кандидата;
- близость считается по стандартизированным признакам;
- пользователи, которые не попали в плотную группу, получают `cluster_id = -1`.

Если DBSCAN не находит хотя бы 2 нормальных кластера, используется fallback:

```text
KMeans
```

Это сделано для того, чтобы при небольшом или слабом датасете программа все равно могла показать возможные группы.

## Как читать `cluster_id`

В `ml_results.csv`:

- `cluster_id = -1` — пользователь не входит в найденную ботоферму;
- `cluster_id = 0`, `1`, `2` и т.д. — пользователь входит в одну из групп бот-кандидатов.

Лучше всего смотреть:

```text
output/analysis_report.html
output/botfarm_graph.html
output/clusters/cluster_*.html
output/bot_candidates.csv
output/cluster_summary.csv
```

## Будут ли данные дублироваться при повторном запуске

Текущая версия использует постоянный кэш и инкрементальный сбор.

При запуске:

```bash
python pipeline.py --all --limit 100
```

программа:

1. читает `data/collection_state.json`;
2. пропускает каналы со статусом `inactive_no_comments` или `invalid_username`, если не указан `--retry-inactive-channels`;
3. собирает новые посты и lookback последних 20 уже виденных постов;
4. добавляет найденные комментарии в `data/comments_cache.csv`;
5. удаляет дубли по стабильному ключу комментария;
6. заново экспортирует рабочий файл `data/raw_comments.csv` из кэша;
7. заново строит features, ML-результаты, графы и отчеты.

То есть старые комментарии не добавляются вторым слоем. Они остаются в `comments_cache.csv`, а новые записи добавляются только если их ключ еще не встречался.

Перед перезаписью проект спрашивает подтверждение:

```text
Overwrite/update generated outputs? [y/N]:
```

Если ответить `y`, старые runtime-файлы будут обновлены. Если не подтвердить, запуск остановится.

### Что защищено от дублей

Есть защита от повторного указания одинаковых каналов:

```text
@truexanewsua,truexanewsua,@truexanewsua
```

будет обработано как один канал:

```text
truexanewsua
```

Есть защита от повторного сбора комментариев:

```text
channel + post_id + comment_id
```

Для старых строк без `comment_id` используется fallback:

```text
channel + post_id + user_id + timestamp + message_text
```

Полный пересбор можно запустить отдельно:

```bash
python pipeline.py --all --limit 100 --full-refresh
```

Он снова пройдет по выбранным каналам, но все равно сольет результат в кэш с дедупликацией.

## Ограничения по сбору комментариев

### 1. Обрабатываются только последние N постов

Параметр `--limit` задает количество последних постов на канал, которые программа проверяет.

Пример:

```bash
python pipeline.py --all --limit 100
```

означает:

```text
проверить до 100 последних постов в каждом канале
```

Это не лимит комментариев. Если у одного поста 500 комментариев, программа может попытаться собрать все доступные комментарии к этому посту.

### 2. Есть программные границы `--limit`

В `config.py`:

```python
MIN_POST_LIMIT = 100
MAX_POST_LIMIT = 500
```

Если указать меньше 100, программа поднимет значение до 100. Если указать больше 500, программа ограничит значение до 500.

Это сделано, чтобы не запускать слишком маленькие или слишком тяжелые сборы.

### 3. Собираются только посты с доступными комментариями

Пост обрабатывается только если Telegram показывает, что у него есть комментарии:

```python
post.replies.comments == True
post.replies.replies > 0
```

Если у канала отключены комментарии или нет связанной discussion group, комментарии собрать не получится.

### 4. Анонимные или недоступные авторы пропускаются

Если у комментария нет `sender_id`, он не сохраняется.

Это может происходить из-за особенностей Telegram, удаленных аккаунтов, анонимности или ограничений доступа.

### 5. Telegram может включать FloodWait

При большом количестве каналов, постов и комментариев Telegram может временно ограничивать запросы.

В таком случае Telethon возвращает `FloodWaitError`, программа ждет указанное Telegram время и спрашивает, продолжать ли сбор.

Пример из логов:

```text
Sleeping for 14s on GetRepliesRequest flood wait
```

Чем больше каналов и чем выше `--limit`, тем чаще возможны такие паузы.

### 6. Доступ зависит от Telegram-аккаунта

Сбор зависит от:

- авторизации Telegram-аккаунта;
- доступности канала;
- наличия комментариев;
- прав доступа к discussion group;
- ограничений Telegram API;
- удаленных или скрытых комментариев.

Поэтому разные аккаунты или разные даты запуска могут дать немного разные результаты.

### 7. Сбор идет по текущему состоянию каналов

Программа берет актуальные последние посты на момент запуска. Если позже появились новые посты или комментарии, следующий запуск может собрать уже другой срез данных.

Это нормальное поведение для динамических социальных сетей.

## Ограничения самой модели

### 1. Нет ручной разметки

Модель не обучалась на заранее размеченных ботах. Она ищет аномалии. Поэтому возможны:

- false positive — активный обычный пользователь выглядит подозрительно;
- false negative — бот ведет себя достаточно естественно и не выделяется.

### 2. `contamination = 0.05` задает ожидаемую долю аномалий

Если в реальности ботов меньше или больше 5%, модель может переоценить или недооценить количество кандидатов.

### 3. Повтор текста ищется простым способом

Levenshtein distance хорошо ловит копипасту и мелкие изменения текста, но плохо ловит смысловые перефразировки.

### 4. Графовая связь зависит от окна 120 секунд

Если ботоферма работает медленнее или делает задержки больше 120 секунд, часть координации может не попасть в граф.

Если сделать окно слишком большим, граф станет шумнее и начнет связывать обычных пользователей.

### 5. Кластеры не равны доказанным ботофермам

Кластер означает, что несколько аномальных аккаунтов похожи по признакам и/или имеют признаки координации. Это сильный сигнал для анализа, но финальный вывод должен учитывать контекст сообщений, каналы, время активности и содержание комментариев.

## Практическая рекомендация для запуска

Для быстрой проверки:

```bash
python pipeline.py --all --limit 100
```

Для более полной курсовой выборки:

```bash
python pipeline.py --all --limit 300
```

Для максимального сбора в текущих ограничениях:

```bash
python pipeline.py --all --limit 500
```

Для перегенерации только отчетов и графов без нового Telegram-сбора:

```bash
python pipeline.py --visualize
```

После запуска сначала открывать:

```text
output/analysis_report.html
```

Затем смотреть:

```text
output/botfarm_graph.html
output/clusters/cluster_*.html
output/bot_candidates.csv
output/cluster_summary.csv
```
